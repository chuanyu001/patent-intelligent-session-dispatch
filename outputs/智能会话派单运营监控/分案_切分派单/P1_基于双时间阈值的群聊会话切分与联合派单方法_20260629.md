## 一、技术背景与现有技术

### 1.1 本发明所处技术领域

本发明属于**计算机信息处理技术领域**，具体涉及一种面向企业即时通信客户群的群聊会话切分与工单派送方法。

### 1.2 现有技术及其缺点

#### 现有技术概述

在企业即时通信（如企业微信）客户群运营场景中，客户在群内以多轮发言的方式提出服务诉求，客服人员需要及时识别这些诉求并分派给合适的处理人员。目前，将群聊消息转化为服务工单的技术方案主要有两类：

**第一类：逐条消息识别派单**（如 CN116032871B——基于聊天消息创建工单的方法、装置和设备）。该类方案获取客户群中的群消息数据，对每条消息或每组消息自动判断是否需要创建工单，若需要则生成工单并发送至工单系统。

**第二类：固定规则派单**（如 CN121814728A——基于企业微信会话存档的客户会话分配与提醒方法及系统）。该类方案按预设排班与分配规则将检测到的新增会话分配给指定处理人员，并内置单级超时检测提醒。

#### 现有技术存在的具体缺点

**缺点1：单句语料不足导致意图误判，多句诉求被错误割裂产生重复工单。**

现有方案（CN116032871B 等）逐条检测消息或整体判断群消息是否需要创建工单，缺乏以会话窗口为单位的聚合与切分机制。实际群聊中，客户的一条完整诉求往往分散在多条消息中（例如"我的车有个问题"→"就是启动的时候有异响"→"大概从上周开始的"），分三条消息发送。逐条处理导致：
- 第一条消息"我的车有个问题"语料不足，大模型无法准确判定意图类别；
- 三条消息被拆成三个独立的工单判断，即使都派单，后两条重复了第一条的诉求；
- 若第一条因语料不足被判定为"闲聊"而丢弃，则后续两条也各自孤悬、同样面临语料不足的问题。

**缺点2：长时间连续对话迟迟不触发派单，形成监控盲区。**

上述方案若仅靠消息间隔（间隔超过阈值才切分），当客户持续快速发言时——例如客户在 20 分钟内连续发送 30 条消息，相邻间隔均小于阈值——会话永远不会被判定"闭合"，工单永远不会生成，形成事实上的监控盲区。CN112686674A（客服会话工单摘要方法）虽按主题边界切分，但为离线摘要场景，不适用于实时派单调度，且同样缺乏时长上限保障。

**缺点3：派单目标固定，无法适配不同群类型、不同意图、不同时段的差异化需求。**

CN121814728A 按排班分配处理人，CN120434222A 按意图路由到工作流，均为单维度固定映射规则。实际运营中：
- 省群（省级服务中心）和店群（门店）的服务角色不同，同一意图应由不同人员处理；
- 同一群在白天工作时段和夜间非工作时段的值班人员不同；
- 特定意图（如备件咨询、保险理赔）需要指定专业部门处理，不能一律派给默认客服。

现有固定映射规则无法同时满足"不同群类型 + 不同意图 + 不同时段"的三维度差异化派单需求。

**缺点4：已派单客户在短时间内继续发言被重复派单，造成多派。**

客户在工单生成后可能继续补充信息（例如工单已派给维修顾问，客户又说"对了还有那个空调也不太制冷"），现有方案未公开抑制窗口机制，会将补充发言判定为新工单再次派发，导致同一客户的同一服务事件被重复派单。

**与总案的区别说明**：上述缺点1、2、3、4 在总案（P0）中作为端到端流程中的两个步骤（S4-S5 会话切分、S13-S16 派单决策）简要描述，总案的技术方案侧重于整体流程组合。本分案将"双阈值判定算法细节"和"三维度优先级决策链"作为独立发明核心展开，公开的算法深度、参数取值策略、数据结构设计远超总案对这两个步骤的描述范围。本分案的技术方案可独立实施，不依赖总案其他步骤的具体实现方式。

## 二、本发明所要解决的技术问题

针对上述缺点，本发明所要解决的技术问题为：

**问题1**：在企业即时通信客户群中，同一客户的多轮发言如何准确判定会话边界——既避免单句语料不足导致意图误判，又避免多句被错误割裂产生重复工单，同时防止长时间连续对话迟迟不触发派单。

**问题2**：如何针对不同群类型、不同意图类别、不同运营时段，自动确定差异化的最优派单目标，克服固定映射规则无法适配多维度差异化需求的缺陷。

**问题3**：如何防止已派单客户在短时间内继续发言时被重复派单，降低多派率。

## 三、技术方案详细阐述

### 3.1 总体技术思路

本发明提出一种**基于双时间阈值的群聊会话切分与联合派单方法**，包含**会话切分阶段**和**联合派单阶段**两个核心阶段：

- **会话切分阶段**：以（群ID + 客户ID）为键维护会话缓冲区，基于"相邻消息间隔阈值 T₁"和"会话持续时长上限 T₂"双时间阈值判定会话边界，生成会话任务。
- **联合派单阶段**：对会话任务的群类型、意图类别、运营时段三个维度进行联合决策，按"时段派送 → 意图指定处理人 → 默认处理人"三级优先级链确定派单目标，并辅以重复抑制窗口防多派。

### 3.2 系统模块架构

本发明的系统包含以下功能模块：

```
┌─────────────────────────────────────────────────┐
│               会话切分子系统                       │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 消息采集  │→│ 会话聚合  │→│ 双阈值判定模块  │  │
│  │ 与过滤   │  │ 缓冲模块  │  │               │  │
│  └──────────┘  └──────────┘  └───────┬───────┘  │
│                                       │          │
│                               会话任务输出        │
├───────────────────────────────────┬─────────────┤
│               联合派单子系统       │             │
│  ┌──────────┐  ┌──────────┐  ┌───┴──────────┐  │
│  │ 意图识别  │→│ 重复抑制  │→│ 联合派单决策  │  │
│  │ 模块     │  │ 模块     │  │ 模块         │  │
│  └──────────┘  └──────────┘  └───────┬──────┘  │
│                                       │          │
│                                 派单目标输出      │
└─────────────────────────────────────────────────┘
```

**消息采集与过滤模块**：对接企业即时通信平台（如企业微信托管账号/机器人），实时采集客户群消息事件，每条消息记录（群ID，客户ID，消息内容，消息时间戳，消息格式）五元组。依据员工名单、机器人名单、客户黑名单过滤非客户诉求消息。

**会话聚合缓冲模块**：以"群ID + 客户ID"为键（Key），为每个键维护一个先进先出（FIFO）的消息缓冲区 B_{g,c} = [m₁, m₂, ..., mₙ]，持续收集该客户在该群内的发言序列。不同客户的消息进入各自独立的缓冲区，互不干扰。

**双阈值判定模块**：对每个缓冲区持续监控两个时间条件，任一条件满足时闭合当前缓冲区、输出会话任务，并清空缓冲区开始新的会话周期。

**意图识别模块**：对输出的会话任务，调用大语言模型进行意图分类，输出母意图 + 子意图 + 置信度三元组。

**重复抑制模块**：维护全局已派单记录表，以（客户ID + 群ID + 意图）为查找键，在新任务生成后、派单前检查是否在抑制窗口 W 内已存在同类工单。

**联合派单决策模块**：对通过抑制检查的会话任务，综合群类型、意图类别、当前时段三个维度，按三级优先级确定派单目标。

### 3.3 详细技术方案——会话切分阶段

#### 3.3.1 数据结构设计

**会话缓冲区定义**：

对于每个键 `(groupID, customerID)`，系统维护一个会话缓冲区 `B`，其数据结构为：

```
B = {
    groupID:     字符串,    // 群标识
    customerID:  字符串,    // 客户标识
    messages:    [m₁, m₂, ..., mₙ],  // 消息列表
    startTime:   时间戳,    // 第一条消息时间 = m₁.timestamp
    lastTime:    时间戳,    // 最后一条消息时间 = mₙ.timestamp
    status:      枚举值,   // "collecting" | "closed"
}
```

其中每条消息 `mᵢ` 包含：
```
mᵢ = {
    msgID:       字符串,
    timestamp:   时间戳（精确到秒）,
    contentType: 枚举值,  // 文字/语音/图片/视频/...
    content:     字符串,  // 消息正文或转写/摘要文本
}
```

#### 3.3.2 双阈值判定算法

系统定义两个独立的时间阈值：

| 阈值符号 | 含义 | 业务作用 |
|----------|------|----------|
| **T₁**（消息间隔阈值） | 相邻两条消息之间的最大允许间隔时间 | 客户沉默超过 T₁，视为一个诉求表达完毕 |
| **T₂**（会话持续时长上限） | 从第一条消息起算的会话最大允许持续时长 | 即使客户持续发言，到 T₂ 也强制闭合，防止长对话永不派单 |

**判定逻辑**：

对于缓冲区 `B` 中的消息序列 [m₁, m₂, ..., mₙ]（n ≥ 1），在以下**任一**条件满足时，闭合该会话缓冲区并输出会话任务 `Task`：

```
条件一（间隔触发）：
    mₙ.timestamp - mₙ₋₁.timestamp ≥ T₁
    含义：最新消息与前一条消息的时间间隔达到 T₁，表明客户已经
          停顿了足够长时间，之前的多轮发言可以视为一次完整的诉求。

条件二（时长触发）：
    mₙ.timestamp - m₁.timestamp ≥ T₂
    含义：从第一条消息到现在已经持续超过 T₂ 时间，即使客户一直
          在密集发言（相邻间隔均 < T₁），也强制输出会话任务，
          避免长对话被无限期搁置。
```

**闭合后处理**：
- 将 B.messages 的全部消息拼接为一个会话任务 Task，输出到下游意图识别模块。
- 清空 B.messages，将 B.startTime 和 B.lastTime 重置为当前消息 mₙ 的时间戳，开始新一轮会话收集。
- B.status 重新设为 "collecting"。

**特殊场景处理**：

1. **缓冲区只有一条消息**（n = 1）：条件一不适用（无前一条消息计算间隔），仅由条件二触发。若 T₂ 时间内无新消息到达，到达 T₂ 后强制闭合输出（单句场景）。

2. **新消息到达时间已经超过 T₁**：即 mₙ 到达时已经满足条件一，则**先闭合之前累积的消息生成任务，再用 mₙ 初始化新的缓冲区**。这样避免了 mₙ 被归入上一段会话（上一段会话该闭合了），也确保了 mₙ 不会丢失（它开启新一轮会话收集）。

3. **T₁ 和 T₂ 同时满足**：以先触发者为准输出会话任务，两者等效。

#### 3.3.3 T₁ 和 T₂ 的取值策略

T₁ 和 T₂ 的取值不是固定的，而是依据群类型和业务场景采用差异化策略：

| 群类型 | T₁ 建议取值 | T₂ 建议取值 | 取值理由 |
|--------|-----------|-----------|----------|
| 技术维修类群 | 8–10 分钟 | 30 分钟 | 问题描述复杂，客户需时间打字、发图，允许较长间隔 |
| 业务咨询类群 | 4–6 分钟 | 20 分钟 | 问答交互快速，短间隔即可完成一轮咨询 |
| 通用群（缺省） | 5 分钟 | 30 分钟 | 兼顾各类场景的平衡配置 |

**实施例中的缺省取值**：T₁ = 300 秒（5 分钟），T₂ = 600 秒（10 分钟）。此配置作为缺省值用于实施例说明，实际部署中可按群类型分别配置，不作为权利要求限制。

#### 3.3.4 双阈值机制的技术效果

与传统单阈值方案相比，双阈值的优势：

| 场景 | 仅用 T₁（间隔阈值） | 双阈值（T₁ + T₂） |
|------|-------------------|-------------------|
| 客户发 3 条消息，间隔均 < T₁ | 正确聚合 ✓ | 正确聚合 ✓ |
| 客户持续快速发言 20 分钟，间隔均 < T₁ | **永远不会触发** ✗ | T₂ 到达后强制触发 ✓ |
| 客户发 1 条消息后无后续 | T₁ 后触发，但语料只有 1 句 | T₂ 后触发，明确知晓是单句场景，可按低置信度处理 |

### 3.4 详细技术方案——联合派单阶段

#### 3.4.1 意图识别与分类体系

会话任务 Task 输出后，调用大语言模型进行意图识别，输出三元组：

```
IntentOutput = (parentIntent, childIntent, confidence)

其中：
- parentIntent ∈ {权益咨询, 销售政策, 售后服务, 信息咨询, 投诉抱怨, 其他类, 闲聊}
- childIntent  ∈ {15 种子意图之一}（如：维修咨询、保养预约、备件查询、保险理赔、……）
- confidence   ∈ [0, 1]，大模型的分类置信度
```

**分类处理规则**：
- `parentIntent = "闲聊"` → 不派单，丢弃该会话任务
- `confidence < θ_conf`（低置信度）或 `parentIntent = "其他类"` → 归入**兜底意图**，仍派单至默认处理人，防止模糊诉求被遗漏
- 其余情况 → 正常进入联合派单决策流程

其中 `θ_conf` 为置信度阈值，实施例取值 0.6，可配置。

#### 3.4.2 重复抑制窗口机制

在进入派单决策前，先检查**重复抑制窗口**，防止同一客户同一诉求被重复派单。

**抑制判定条件**：

对于新会话任务 Task_new，当以下条件**全部**满足时，判定为重复，不再生成新工单而是合并到已有工单 Ticket_existing：

```
条件A：Task_new.groupID == Ticket_existing.groupID       （同一群）
条件B：Task_new.customerID == Ticket_existing.customerID  （同一客户）
条件C：Task_new.parentIntent == Ticket_existing.parentIntent
       OR sim(Task_new, Ticket_existing) ≥ θ           （意图相同或事件相似）
条件D：Task_new.timestamp - Ticket_existing.createTime ≤ W （抑制窗口内）
```

**抑制窗口 W 的取值**：

| 参数 | 含义 | 实施例缺省值 |
|------|------|------------|
| W | 同一客户同一诉求不重复派单的时间窗口 | 1800 秒（30 分钟） |

实施例：派单后 30 分钟内，同一客户在同一群的继续留言（意图相同或事件相似），合并到已有工单，不重复派单。

**合并处理方式**：
- 将 Task_new 的消息内容追加到 Ticket_existing 的备注/补充信息字段
- 更新 Ticket_existing 的最后活跃时间
- 可选择向处理人员发送"客户补充了新信息"的通知

#### 3.4.3 三维度联合派单决策

对通过抑制检查的会话任务，按**群类型 × 意图类别 × 运营时段**三个维度联合决策派单目标。

**三个维度的定义**：

| 维度 | 取值 | 说明 |
|------|------|------|
| 群类型 G_type | 省群 / 店群 / 一对一群 / 合作伙伴群 | 预先在群管理中为每个群配置 |
| 意图类别 I_cat | 7 类母意图 | 由意图识别模块输出 |
| 运营时段 T_zone | 工作日白天 / 工作日夜间 / 非工作日白天 / 非工作日夜间 | 按当前系统时间自动判定 |

**三级优先级决策链**：

```
优先级 1 → 时段派送映射
    ↓ （返回空）
优先级 2 → 意图指定处理人映射
    ↓ （返回空）
优先级 3 → 默认处理人映射
    ↓
输出派单目标 Dest
```

**第一级：时段派送映射** `f_zone(T_zone, G_type)`

根据当前时段和群类型，查表获取时段排班人员配置。

配置示例（不作为权利要求限制）：

| 时段 | 省群处理人 | 店群处理人 |
|------|----------|----------|
| 工作日白天 (09:00-18:00) | 省客服团队企微号 | 门店客服企微号 |
| 工作日夜间 (18:00-09:00) | 值夜班企微号 | 门店值班企微号 |
| 非工作日全天 | 值班企微号 | 门店值班企微号 |

若查询到非空结果（即当前时段配置了处理人），则直接将 Dest 设为该结果，**不再执行后续优先级**。

**第二级：意图指定处理人映射** `f_intent(I_cat, G_type)`

当第一级返回空时（即当前时段未配置排班），根据意图类别和群类型查表获取指定处理人。

配置示例（不作为权利要求限制）：

| 意图类别 | 省群指定处理人 | 店群指定处理人 |
|----------|-------------|-------------|
| 备件咨询 | 备件部企微号 | 门店备件专员 |
| 保险理赔 | 保险部企微号 + 400 客服（一单多派） | 门店保险专员 |
| 投诉抱怨 | 客服主管企微号 + 400 客服群（一单多派） | 店长企微号 |

**第三级：默认处理人映射** `f_default(G_type)`

当前两级均返回空时，取群类型的默认处理人。

配置示例：

| 群类型 | 默认处理人 |
|--------|----------|
| 省群 | 省客服默认企微号 |
| 店群 | 门店默认客服企微号 |
| 一对一群 | 群主企微号 |

**派单目标的形式**：
- **单目标**：一个企微号，生成一条工单推送给该号
- **多目标（一单多派）**：多个企微号或企微群，生成一条工单同时推送给多个目标（如投诉意图同时派给客服主管和 400 客服群）

#### 3.4.4 一单多派的触发条件

当满足以下任一条件时，Dest 包含多个目标（一单多派）：

| 触发条件 | 示例 |
|----------|------|
| 意图为高风险类别（投诉抱怨、重大故障） | 同时派主管 + 400 客服群 |
| 意图涉及跨部门协调 | 同时派备件部 + 维修部 |
| 群类型为省群且意图为升级类 | 同时派省客服 + 总部质检 |

一单多派时，所有目标收到的工单共享同一工单 ID，任一目标回复后工单状态同步更新。

### 3.5 端到端主流程

```
步骤 S1：采集群消息事件（群ID + 客户ID + 内容 + 时间戳 + 格式）
        ↓
步骤 S2：过滤员工/机器人/黑名单消息
        ↓
步骤 S3：会话聚合 → 按（群ID + 客户ID）键写入对应缓冲区 B_{g,c}
        ↓
步骤 S4：双阈值判定
        条件一：相邻消息间隔 ≥ T₁ ？
        条件二：会话持续时长 ≥ T₂ ？
        ├── 均不满足 → 继续 S3 收集
        └── 任一满足 → 闭合缓冲区，输出会话任务 Task
        ↓
步骤 S5：意图识别 → (parentIntent, childIntent, confidence)
        ├── 闲聊 → 丢弃
        └── 非闲聊 → 进入 S6
        ↓
步骤 S6：重复抑制检查
        同一（客户 + 群 + 意图/相似事件）在抑制窗口 W 内已有工单？
        ├── 是 → 合并到已有工单（不重复派单）
        └── 否 → 进入 S7
        ↓
步骤 S7：三维度联合派单决策
        Dest = 第一非空（时段派送映射, 意图指定映射, 默认映射）
        ↓
步骤 S8：生成工单并推送至 Dest（单目标或多目标）
        ↓
步骤 S9：记录派单时间、更新全局已派单记录表（供 S6 抑制检查用）
```

### 3.6 关键技术参数汇总

| 符号 | 参数含义 | 实施例缺省值 | 可配置性 |
|------|----------|------------|---------|
| T₁ | 相邻消息间隔阈值 | 300 秒（5 分钟） | 按群类型可配置 |
| T₂ | 会话持续时长上限 | 600 秒（10 分钟） | 按群类型可配置 |
| W | 重复抑制窗口 | 1800 秒（30 分钟） | 全局可配置 |
| θ | 事件相似度阈值（抑制判定用） | 0.7 | 全局可配置 |
| θ_conf | 意图置信度阈值（兜底判定用） | 0.6 | 全局可配置 |

上述取值仅为实施例说明，不作为权利要求限制。

## 四、与现有技术相比的优点

**优点1：双阈值会话切分机制解决了群聊场景中"单句语料少、多句易割裂、长对话不派单"三个子问题。**

T₁ 将客户的多轮发言自动聚合为完整会话，克服单句语料不足的缺陷；T₂ 提供时长上限保障，即使客户持续密集发言也强制在 T₂ 后输出任务，消除监控盲区。现有技术（CN116032871B、CN121480730A 等）均未公开 T₁ + T₂ 双阈值组合的会话切分方法。

**优点2：三维度联合派单 + 三级优先级决策链实现了差异化的动态派单。**

综合群类型、意图类别、运营时段三个维度，按"时段优先 → 意图指定 → 默认兜底"三级优先级自动匹配最优处理人。相比 CN121814728A 的固定排班分配、CN120434222A 的单维度意图路由，本发明可适配省群/店群/一对一群的角色差异、工作/非工作时段的值班差异、不同意图的专业处理人差异。

**优点3：重复抑制窗口 W 有效降低多派率。**

通过在 W 窗口内合并且不重复派单，解决了客户补充发言导致的重复工单问题。现有检索范围内无任何专利公开此机制。

**优点4：兜底意图 + 置信度判定防止漏单。**

对低置信度意图和其他类意图不丢弃，仍派单至默认处理人，确保模糊诉求不被完全遗漏。

**优点5：本方案可独立实施，不依赖总案其他模块。**

会话切分模块和联合派单模块构成完整的"消息 → 工单"处理链路，只需接入消息源和意图识别服务即可独立运行。

## 五、技术关键点和欲保护点

1. **双阈值会话任务生成方法**：以群标识与客户标识为键维护会话缓冲区；以相邻消息间隔阈值 T₁ 和会话持续时长上限 T₂ 双条件判定会话边界——任一条满足即闭合缓冲区、输出会话任务（参见 3.3.2 节判定算法）。

2. **T₁ 和 T₂ 的差异化取值策略**：按群类型（技术维修类、业务咨询类、通用类等）分别设置不同的 T₁/T₂ 取值，适应不同场景的消息交互节奏（参见 3.3.3 节）。

3. **重复抑制窗口机制**：已派单用户在抑制窗口 W 内的同群同意图/相似事件继续留言，合并到已有工单、不重复派单（参见 3.4.2 节判定条件）。

4. **三维度联合派单决策方法**：综合群类型、意图类别、运营时段三个维度，按"时段派送 → 意图指定处理人 → 默认处理人"三级优先级确定派单目标（参见 3.4.3 节决策链）。

5. **一单多派触发规则**：当意图为高风险类别或涉及跨部门协调时，单条工单同时派送至多个目标，共享工单 ID 同步状态（参见 3.4.4 节）。

6. **兜底意图防漏单机制**：低置信度意图或其他类意图不丢弃，归入兜底意图仍派单至默认处理人（参见 3.4.1 节）。

## 六、建议权利要求框架

### 独立权利要求 1（方法——切分+派单两段式）

一种基于双时间阈值的群聊会话切分与联合派单方法，其特征在于，包括：

**会话切分阶段**：
（a）实时采集企业即时通信客户群中的消息事件，提取群标识、客户标识、消息内容和时间戳；
（b）以群标识与客户标识为键，维护会话消息缓冲区，持续收集同一客户在同一群内的消息序列；
（c）当满足第一条件——最新消息与前一条消息的时间间隔达到或超过第一阈值 T₁，或满足第二条件——缓冲区中第一条消息与最新消息的时间间隔达到或超过第二阈值 T₂——时，闭合该缓冲区并将会话内容整体输出为一个会话任务；
（d）若缓冲区仅含一条消息且未达到 T₂，继续等待后续消息直至满足第（c）步条件；

**联合派单阶段**：
（e）对会话任务进行意图识别，输出意图类别；
（f）在生成新工单前，检查是否存在同一客户在同一群内、意图相同或近似且创建时间在抑制窗口 W 内的已有工单——若存在则合并到该已有工单，不重复派单；
（g）对通过抑制检查的会话任务，依据群类型、意图类别、当前运营时段三个维度，按预设的优先级规则确定派单处理人员；
（h）将派单处理人员信息写入工单并推送。

### 独立权利要求 2（系统）

一种基于双时间阈值的群聊会话切分与联合派单系统，其特征在于，包含用于执行权利要求 1 所述方法的各步骤的模块。

### 从属权利要求建议

## 七、实施例

### 实施例 1：单人单句场景

**场景描述**：客户在省群中发送一条消息"我的车启动的时候有异响，咋回事"，之后无后续消息。

**处理过程**：
1. 系统采集该消息，写入（省群ID，客户A）的会话缓冲区 B，startTime = lastTime = 当前时间。
2. 由于缓冲区仅含 1 条消息，无前一条消息无法计算间隔，仅监控 T₂。
3. T₂ = 600 秒内无新消息到达 → 600 秒后 T₂ 条件触发，闭合 B，输出会话任务 Task。
4. Task 仅有 1 条消息，送入意图识别 → (售后服务, 维修咨询, 0.72)。
5. 联合派单：当前为工作日白天 → 时段派送返回"省客服团队企微号" → 派单至该号。

**效果**：单条消息虽然语料少，但能在 T₂ 后及时派单，置信度 0.72 > θ_conf(0.6)，正常派单。

### 实施例 2：单人多句场景（间隔 < T₁，被正确聚合）

**场景描述**：同一客户在 3 分钟内连发 3 条消息：

| 时间 | 消息内容 |
|------|----------|
| t₀ + 0s | "我的车启动的时候有异响" |
| t₀ + 60s | "就是那种嘎吱嘎吱的声音" |
| t₀ + 150s | "大概从上周开始出现的" |

**处理过程**：
1. 第一条消息到达，初始化缓冲区 B，startTime = t₀。
2. 第二条消息到达，间隔 60s < T₁(300s)，追加到 B。
3. 第三条消息到达，间隔 90s < T₁(300s)，追加到 B。
4. 客户端没有再发新消息。450s 后（从第三条消息起算），间隔达到 T₁(300s)，条件一触发，闭合 B。
5. 输出 Task，包含全部 3 条消息："我的车启动的时候有异响 / 就是那种嘎吱嘎吱的声音 / 大概从上周开始出现的"。
6. 意图识别有充足的上下文 → (售后服务, 维修咨询, 0.89)，置信度明显高于实施例 1。

**对比现有技术**：若逐条识别，三条消息分别需独立判断，可能得出三个不同的低置信度结果；本方案聚合后输出一次，语料充足、置信度高，且只生成一个工单。

### 实施例 3：长时间连续发言场景（T₂ 强制触发）

**场景描述**：客户在 25 分钟内连续发送 17 条消息，每条间隔约 80-100 秒（均 < T₁）。

**处理过程**：
1. 客户持续发言，缓冲区不断增长，相邻间隔始终 < T₁(300s)，条件一永不满足。
2. 在第 10 分钟（600s）时，T₂ 条件触发 → 闭合缓冲区，输出任务。
3. 注意：此时客户还在继续发言。T₂ 触发时闭合的是**当前已有的消息**（第 1 条到约第 6-7 条），闭合后缓冲区清空，后续消息开启新一轮会话收集。

**对比现有技术**：若仅有间隔阈值 T₁，此场景下会话永远不会闭合，17 条消息在 25 分钟内全部堆积在缓冲区中，工单迟迟不生成，形成监控盲区。T₂ 的引入确保最长 600 秒一定输出一次任务。

### 实施例 4：重复抑制窗口效果

**场景描述**：客户 A 的会话任务已生成工单 Ticket_001（意图：售后服务-维修咨询），派单时间 14:00。5 分钟后（14:05），客户 A 在同一群又发消息"对了，空调也不制冷"，触发新的会话任务 Task_2。

**处理过程**：
1. Task_2 意图识别：(售后服务, 维修咨询, 0.81)。
2. 重复抑制检查：同一群、同一客户、同一意图、14:05 - 14:00 = 300s < W(1800s)。
3. 判定为重复 → 合并到 Ticket_001，追加"空调也不制冷"为补充信息，不生成新工单。

**效果**：避免同一服务事件被拆分派发为两个工单，降低多派率。

### 实施例 5：三维度联合派单——时段优先

**场景描述**：某省群客户在周六晚上 21:00 发起会话，意图为"备件咨询"。

**处理过程**：
1. 意图识别：(备件咨询, 某备件查询, 0.85)。
2. 联合派单决策：
   - 第一级（时段）：当前为非工作日夜间 → 查表返回"省群值班企微号"。
   - Dest = 省群值班企微号，不再执行后续优先级。
3. 工单推送给值班人员。

**对比**：若没有时段维度，该工单会按意图映射派给"备件部企微号"（但备件部此时无人值班），或按默认派给"省客服默认企微号"（可能也无人值守）。时段优先级确保非工作时间自动切换至值班人员。

### 实施例 6：测试数据验证

以某售后服务客户大群的实际运营数据为实施例（数据已脱敏）：

- **测试周期**：覆盖一定时长，累计监听会话约 12448 条
- **客户会话**：约 8954 条
- **生成工单**：1227 条
- **核心指标**：工单准确率 > 90%，漏单率 0%，多派率 < 30%
- **参数配置**：T₁ = 300s、T₂ = 600s、W = 1800s、θ = 0.7、θ_conf = 0.6
- **覆盖场景**：单人单句、单人单句II（类闲聊被正确丢弃）、单人多句、多人多句（含多人聚合、多人交叉互助）4 种场景
