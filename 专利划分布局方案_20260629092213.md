## 一、产品技术全景


### P1 · 分案A：一种基于双时间阈值的群聊会话切分与联合派单方法

| 项目 | 内容 |
|------|------|
| **技术问题** | （1）群聊中同一客户的多轮发言如何判定会话边界——单句语料少、多句易割裂、长对话迟迟不触发；（2）如何对不同群类型、不同意图、不同时段实施差异化派单 |
| **技术方案** | 以「群ID+客户ID」为键维护会话缓冲区；相邻消息间隔超 T₁ 或会话持续时长超 T₂ 时触发会话任务输出（式 1）；按「时段派送 > 指定意图处理人员 > 默认处理人员」三级优先级联合决策派单目标（式 2a–2c） |
| **区别于总案的重点** | 总案中这两个点作为流程中的两个步骤一笔带过；分案将**双阈值判定**和**优先级决策链**作为核心独立展开，可写入更细的实施例（如不同 T₁/T₂ 取值策略、三级决策的配置界面、优先级冲突消解规则） |
| **现有技术区分** | 现有方案（CN122221843A）以消息标签匹配调度，未公开同群同客户双阈值切分；现有派单为固定映射，未公开三维度优先级决策链 |
| **覆盖的技术点** | ①双阈值会话切分 ②联合派单决策 ③重复抑制窗口 ④兜底意图派单 ⑤一单多派（高风险外升级的派单维度） |
| **建议权利要求** | 方法独立权利要求（切分+派单两段式）+ 系统权利要求 + 切分子方法 + 派单决策子方法 |
| **实施例素材** | 测试报告中 4 种场景（单人单句/单人单句II/单人多句/多人多句）的会话分布数据、5min/10min/30min 三参数配置 |

### P2 · 分案B：一种派单与监控解耦的企业通信群分级报警方法

| 项目 | 内容 |
|------|------|
| **技术问题** | 现有方案中派单与监控紧耦合——关闭派单即关闭监控，运营方无法在不对外派单的情况下保留内部质检记录；超时报警为单一阈值，无法区分不同风险等级 |
| **技术方案** | 工单派送开关与超时监控开关**独立控制**，关闭派送后系统仍生成工单记录（标记为不派单）并继续监控；建立**三级并行**监控任务——轻度超时（T_mild）、严重超时（T_severe）、意图报警（T_intent，T_intent < T_mild）；各开关可独立启停；报警可按意图类别、群类型、时段分别配置 |
| **区别于总案的重点** | 总案中监控作为流程末尾环节简要描述；分案将**开关解耦的电路/逻辑设计**和**三级报警的状态机**作为独立发明详述 |
| **现有技术区分** | 现有群聊消息处理和 SCRM 产品的派单与监控均为紧耦合；未公开独立开关控制+三级并行监控的组合架构 |
| **覆盖的技术点** | ①派单/监控开关解耦 ②轻度超时报警 ③严重超时报警 ④意图专属报警（含缩短阈值）⑤按时段/群类型的差异化报警配置 |
| **建议权利要求** | 监控方法独立权利要求 + 系统权利要求 + 开关解耦装置权利要求 + 三级报警状态机 |
| **实施例素材** | 测试报告的运营监控配置（超时监听时长/轻度超时阈值/严重超时阈值/超时报警开关/工单派送开关/意图监听时长/识别意图开关） |

### P3 · 分案C：一种面向企业通信群的多格式消息处理与意图分类方法

| 项目 | 内容 |
|------|------|
| **技术问题** | （1）企微客户群消息格式多达 12 种，客户常用语音、图片描述故障，传统文本工单易漏非文本诉求；（2）通用语音识别对行业术语、方言、简称识别错误率高，导致意图分类偏差；（3）模糊意图容易被完全丢弃造成漏单 |
| **技术方案** | 将 12 种消息格式按处理方式分为三类——Ⅰ类（文字/语音→按意图分类派单）、Ⅱ类（图片/视频/链接/文件/小程序→提取内容摘要后归入图片类派单）、Ⅲ类（表情包/撤回/红包/位置/卡片→仅识别不处理）；语音转文字后经**行业领域词库**模糊匹配纠错（相似度 > λ 替换为规范术语，≤ λ 标记低置信度供人工复核）；大模型输出母意图+子意图+风险等级，闲聊不派单，兜底意图捕获模糊诉求防止漏单 |
| **区别于总案的重点** | 总案中归一化和分类作为流程中的一个模块简述；分案将**12 种格式的分类处理规则**、**领域词库的构建与匹配算法**、**兜底意图的设计逻辑**独立展开 |
| **现有技术区分** | CN120434222A/CN119917631A 仅处理文字消息意图识别，未涉及 12 种格式的差异化处理；现有语音识别无行业领域词库纠错环节 |
| **覆盖的技术点** | ①12 种格式三分类规则 ②语音转写+领域词库纠错 ③Ⅱ类格式内容提取与归并 ④母意图+子意图+风险等级三层输出 ⑤兜底意图防漏单 |
| **建议权利要求** | 消息处理方法独立权利要求 + 语音纠错子方法 + 领域词库构建方法 + 系统权利要求 |
| **实施例素材** | 测试报告中「会话格式」表格（12 种格式及处理方式）、「SP→备件」纠错案例、7 类母意图/15 类子意图体系 |

### P4 · 分案D：一种基于多指标的派单策略闭环优化方法及系统

| 项目 | 内容 |
|------|------|
| **技术问题** | 现有方案以意图识别准确率作为单一优化目标，单纯提升准确率可能导致漏单率上升或多派率恶化；系统上线后派单策略参数为静态配置，无法随业务变化自动适应 |
| **技术方案** | 采集每条会话的原始消息、意图分类结果、派单目标、人工复核结论和响应时长；统计**工单准确率 A、漏单率 L、多派率 M** 三项指标；当 A < 90% 或 L > 1% 或 M > 30% 时自动触发策略参数调整——包括调整会话切分阈值 T₁/T₂、重复抑制窗口 W、意图分类置信度 θ_conf、兜底派单触发条件；调整后持续监控三项指标的联动变化，避免单指标优化导致其他指标恶化 |
| **区别于总案的重点** | 总案中反馈作为流程末尾 S16–S17 简要描述；分案将**三指标的定义与计算方法**、**阈值触发逻辑**、**参数调整算法**、**指标联动约束**独立详述 |
| **现有技术区分** | 现有方案以单指标（准确率）为优化目标；未公开基于三指标协同约束的自动调参方法 |
| **覆盖的技术点** | ①三指标定义与计算公式 ②指标采集与人工复核流程 ③阈值自动触发逻辑 ④参数调整算法（含指标联动约束）⑤调参效果验证与回滚 |
| **建议权利要求** | 策略优化方法独立权利要求 + 三指标计算方法 + 参数调整决策方法 + 系统权利要求 |
| **实施例素材** | 测试报告的 12448 条会话/1227 条工单/准确率>90%/漏单率0%/多派率<30%，以及错派/漏派/多派原因分析 |

## 四、分案与总案的关系

| 维度 | 说明 |
|------|------|
| **优先权** | 5 件同日申请，均主张同一优先权；或先提交总案，分案在 12 个月内以总案为优先权基础提交 |
| **重复授权** | 分案权利要求与总案**不实质相同**——总案保护整体流程组合，分案各保护一个子系统的深度实现；即使总案权利要求覆盖某步骤，分案对该步骤的具体算法/结构/参数的描述远超总案公开范围 |
| **说明书共享** | 分案说明书可引用总案的整体架构作为背景，重点展开各自的核心技术细节；实施例侧重各自领域 |
| **侵权举证** | 总案适合抓「全流程抄袭」；分案适合抓「只抄某个子系统」（如只抄监控模块、只抄切分逻辑），降低举证难度 |

## 七、建议的申请策略

1. **同日提交**：5 件同日提交，互为印证，增加整体授权概率。
2. **优先权链**：若资源有限，先提交 P0+P1，12 个月内以 P0 为优先权提交 P2/P3/P4。
3. **说明书篇幅**：每件分案 15–25 页为宜，实施例至少 3 个场景。
4. **权利要求数量**：每件 10 项左右（方法独立项 1–2 + 系统独立项 1 + 从属项 7–8）。
5. **附图独立**：每件分案绘制专属附图，不与总案共用同一张图。
